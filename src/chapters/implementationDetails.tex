\section{Dataset Extraction and Preparation}

\subsection{Data Sources}

Two complementary datasets were collected from publicly available environmental data
repositories: historical air quality observations from the OpenAQ v3 API, and
meteorological observations from the Open-Meteo Historical Weather API. The two
datasets were selected because atmospheric conditions play a significant role in the
transport, dispersion, and accumulation of air pollutants.

\textbf{Air Quality Dataset:} The air quality observations were collected from the
OpenAQ v3 API, an open platform that aggregates measurements from ground-based air
quality monitoring stations. Station discovery was performed automatically by querying
all monitoring stations located within a 25 km radius of central Kathmandu (Latitude:
27.7172$^\circ$, Longitude: 85.3240$^\circ$). For every discovered station, the
corresponding PM$_{2.5}$ sensor ID was extracted, and hourly PM$_{2.5}$ concentration
values with timestamps were downloaded.

\textbf{Meteorological Dataset:} Historical meteorological estimates were obtained from the Open-Meteo Historical Weather API. These variables are derived from numerical weather reanalysis and model products rather than from meteorological instruments installed at each air-quality station. Using the latitude and longitude of each monitoring station, hourly weather observations were downloaded individually. Because the meteorological data have a coarser spatial resolution than the air-quality monitoring network, local wind and terrain effects may not be fully represented. The number of unique weather grid cells used across the selected stations will therefore be examined before graph construction.

\subsection{Dataset Extraction Pipeline}

The dataset extraction process was implemented as an automated, modular pipeline
consisting of several independent scripts. Each script performed a specific task and
produced an intermediate output that serves as the input for the next stage.

Each stage is implemented as an independent Python module, and all stages are
orchestrated by a single driver script that runs every stage in order and halts at the
first failing stage, allowing the pipeline to be executed either step-by-step or
automatically end-to-end.

\subsection{Dataset Description and Organization}

The collected dataset consists of hourly observations from multiple air quality
monitoring stations distributed across the Kathmandu Valley. Each observation
corresponds to a single timestamp at a particular monitoring station and contains both
pollutant concentrations and the associated weather conditions. Since the data are
collected independently for each station, the complete dataset is represented as
multiple station-wise CSV files, where each file stores the temporal observations of a
single monitoring station.

In addition to the station-wise observations, a separate station metadata file is
maintained containing the geographical information of every monitoring station,
including its latitude and longitude, and serves as the basis for subsequent graph
construction. The overall dataset therefore consists of:

\begin{itemize}
    \item Station-wise air quality and meteorological observations (49 stations with usable PM$_{2.5}$ records)
    \item Station metadata containing geographical information
    \item Common hourly timestamps for temporal synchronization
\end{itemize}

\subsection{Feature Description}

The collected datasets contain both air quality measurements and meteorological
observations. Each variable contributes different information required for
forecasting future PM$_{2.5}$ concentrations.

\textbf{Air Quality Features:} PM$_{2.5}$ (prediction target), PM1 (recorded where
available, though not present in the final collected dataset used for modeling).

\textbf{Meteorological Features:} temperature, relative humidity, dew point, surface
pressure, wind speed, wind direction, precipitation.

Among these features, PM$_{2.5}$ is used as the prediction target, while the remaining
meteorological variables serve as explanatory input features. The spatial attributes
(latitude and longitude) are used during graph construction rather than as direct
prediction inputs. Following preprocessing, additional engineered variables -- lag
features, rolling statistics, cyclical time encodings, and wind vector components --
were generated to improve the predictive capability of the forecasting models; these
are described in Section 5.5.

\section{Data Preprocessing}

The raw datasets collected from OpenAQ and Open-Meteo cannot be used directly for
machine learning because they originate from different sources, have different
structures, and contain inconsistencies such as missing timestamps or incomplete
measurements. A systematic preprocessing pipeline was therefore developed to transform
the raw data into a clean, synchronized, model-ready dataset.

\subsection{Dataset Validation}

After downloading the weather and air-quality datasets, an automated validation process
was performed to verify the integrity and consistency of the collected data. Since the
project contains multiple CSV files across different stations and years, the
validation process automatically inspects every file and generates summary reports.
For every dataset, the validation script checks:

\begin{itemize}
    \item File readability (corrupted or valid)
    \item Number of rows and columns
    \item Duplicate timestamps
    \item Missing values
    \item First and last timestamps
    \item Chronological ordering of timestamps
\end{itemize}

Based on these checks, two validation reports -- \texttt{weather\_validation.csv} and
\texttt{air\_quality\_validation.csv} -- are generated, summarizing the quality of each
dataset before proceeding to preprocessing.

\subsection{Timestamp Normalization}

The weather and air-quality datasets originate from two different APIs, and although
both provide hourly observations, their timestamp formats are not always identical.
Timestamps were therefore standardized before merging. During this stage:

\begin{itemize}
    \item Timestamps were converted into a common datetime format
    \item Timezone inconsistencies were removed
    \item Timestamps were sorted chronologically
    \item Duplicate timestamps were eliminated
\end{itemize}

After normalization, every observation followed the same temporal representation,
allowing accurate alignment between weather variables and PM$_{2.5}$ measurements.

\subsection{Dataset Synchronization and Merging}

Since weather and air-quality observations are collected independently, they must be
synchronized on their standardized timestamps before model training. For each
monitoring station, air-quality data is left-joined onto the weather timeline
(weather as the base table), since Open-Meteo's historical record is continuous at
hourly resolution. This guarantees one row per hour in the merged dataset, with
\texttt{pm2\_5} left as \texttt{NaN} wherever no reading exists (a property later
relied upon for row-position-based feature engineering).

The merged dataset therefore contains:

\begin{itemize}
    \item PM$_{2.5}$ concentration
    \item Temperature
    \item Relative humidity
    \item Dew point
    \item Atmospheric pressure
    \item Wind speed
    \item Wind direction
    \item Precipitation
    \item Station information
    \item Geographical coordinates
\end{itemize}

This merged dataset forms the primary input for subsequent preprocessing and feature
engineering stages.

\subsection{Removal of Initial Missing PM$_{2.5}$ Records}

After merging, some stations contain long initial periods where weather observations
are available but PM$_{2.5}$ measurements have not yet started. These rows do not
contribute useful information for forecasting, since the target variable is absent.
This trimming process:

\begin{itemize}
    \item Removes unnecessary records
    \item Reduces storage requirements
    \item Ensures that every remaining observation contains the target pollutant
    \item Simplifies later feature generation
\end{itemize}

Only leading missing records were removed; missing values occurring later in a
station's timeline were preserved for the target-invalidation logic described in
Section 5.7.

\subsection{Timestamp Continuity Validation}

After merging and trimming -- and \textit{before} feature engineering -- an additional
verification step confirms that the temporal sequence remains continuous. This stage
checks:

\begin{itemize}
    \item Hourly interval consistency
    \item Chronological ordering
    \item Missing timestamps
    \item Duplicated timestamps
    \item Unexpected temporal gaps
\end{itemize}

Maintaining correct temporal ordering at this stage is particularly important because
forecasting models such as LSTM, GRU, and Graph Neural Networks rely on sequential
dependencies between observations, and the lag/rolling features generated in the next
stage rely on row position corresponding to real elapsed time. Any inconsistencies
detected are reported automatically per station.

\subsection{Feature Engineering}

Once timestamp continuity was verified, additional predictive features were generated
per station:

\textit{Lag Features.} Historical PM$_{2.5}$ observations are introduced as predictor
variables to capture temporal dependency, allowing the forecasting models to utilize
recent pollution history: \texttt{lag\_1}, \texttt{lag\_3}, \texttt{lag\_6},
\texttt{lag\_12}, \texttt{lag\_24} (PM$_{2.5}$ at 1, 3, 6, 12, and 24 hours prior).

\textit{Rolling Statistical Features.} Moving statistical measures summarize recent
pollutant behavior over fixed windows, capturing short-term pollution trends and
variability: rolling mean and rolling standard deviation over 3-, 6-, and 24-hour
windows.

\textit{Cyclical Time Features.} Time variables such as hour of day and month of year
exhibit periodic behavior and are therefore encoded using sine/cosine pairs
(\texttt{hour\_sin}/\texttt{hour\_cos}, \texttt{month\_sin}/\texttt{month\_cos}) rather
than raw integers. This encoding preserves the cyclic nature of these variables and
avoids an artificial discontinuity between, e.g., hour 23 and hour 0. \textit{A
cyclical encoding of day-of-week was considered during planning but is not present in
the current implementation; the raw \texttt{weekday} column is retained instead and is
a candidate for future refinement.}

\textit{Wind Vector Components.} Rather than using wind speed and wind direction
independently, they are transformed into Cartesian vector components:
$$wind_u = WindSpeed \cdot \cos(\theta), \qquad wind_v = WindSpeed \cdot \sin(\theta)$$
where $\theta$ is the wind direction. This representation preserves both wind magnitude
and direction, is more suitable for machine learning models than the raw polar form,
and provides the foundation for the wind-aware graph construction described in
Section 5.9.

\subsection{Dataset Preparation and Target Definition}

Following feature engineering, the dataset was transformed into its final
machine-learning-ready form. This stage includes selecting the relevant predictor
variables, defining the PM$_{2.5}$ prediction target, removing unused attributes, and
arranging columns into a consistent format.

The prediction target itself is defined as the PM$_{2.5}$ concentration one hour ahead
of the current observation: $target\_pm2\_5(t) = pm2\_5(t+1)$. A naive
\texttt{.shift(-1)} implementation of this is unsafe in the presence of timestamp gaps,
since it grabs whichever row happens to sit next in the file, regardless of how much
real time separates it from the current row. To prevent this, the target is
invalidated whenever the true gap to the next row is not exactly one hour:

\begin{verbatim}
next_timestamp = df["timestamp"].shift(-1)
time_gap = next_timestamp - df["timestamp"]

df["target_pm2_5"] = df["pm2_5"].shift(-1)
df.loc[time_gap != pd.Timedelta(hours=1), 
"target_pm2_5"] = pd.NA
\end{verbatim}

Rows with an invalidated or otherwise missing target are dropped, and the number of
rows removed is logged per station. At the end of this stage, each observation
contains all engineered features together with a reliably-labeled target PM$_{2.5}$
value.

\subsection{Train / Validation / Test Split}

The prepared dataset was divided into three mutually exclusive, chronologically
ordered subsets per station: training, validation, and testing. Chronological (rather
than random) splitting was adopted to prevent information leakage from future
observations into past data and to reflect a realistic forecasting scenario. The
training dataset is used for model learning, the validation dataset for hyperparameter
tuning, and the testing dataset for final performance evaluation.

\subsection{Split Verification}

An automated verification process confirms that the generated datasets satisfy the
required conditions:

\begin{itemize}
    \item Correct number of samples
    \item Chronological ordering
    \item Absence of overlapping timestamps between subsets
    \item Successful separation of training, validation, and testing data
    \item Sufficient observations for model training
\end{itemize}

Only after successful verification are the datasets considered ready for baseline model
training (Persistence and Ridge Regression).

\section{Baseline Forecasting Models}

\subsection{Persistence Baseline}

Predicts next-hour PM$_{2.5}$ as equal to the current reading:
$\hat{y}_{t+1} = pm2\_5(t)$. Despite its simplicity, this is a strong baseline for
short-horizon PM$_{2.5}$ forecasting, since pollution levels change relatively little
hour to hour.

\subsection{Linear (Ridge) Regression Baseline}

\textbf{Initial attempt and multicollinearity issue:} The first implementation used
ordinary least squares (OLS) with every engineered column as input. This produced an
average test $R^2$ of -2.98, substantially worse than persistence, with some station
predictions falling far outside the physically plausible PM$_{2.5}$ range.

\section{Wind-Aware Graph Construction (In Progress)}

Development of the wind-aware dynamic graph module has begun as a separate pipeline.

\textbf{Completed:}
\begin{itemize}
    \item \textbf{Station node mapping} -- assigns each station a numeric graph node ID from its metadata, producing a station-to-ID lookup used by all subsequent graph steps.
    \item \textbf{Distance matrix generation} -- computes the pairwise Haversine (great-circle) distance between every pair of stations, producing a full distance matrix and a flattened edge list. This implements the spatial-proximity term $f_d(i,j)$.
\end{itemize}

\textbf{Remaining:}
\begin{itemize}
    \item Bearing matrix (compass direction between station pairs, needed to compare against wind direction)
    \item Static graph assembly (distance + historical PM$_{2.5}$ correlation)
    \item Dynamic wind-aware edge weights (time-varying term $f_w$)
    \item Graph snapshot generation (one adjacency matrix per timestamp)
    \item Sliding window construction for temporal model input
\end{itemize}
