\section{Dataset Extraction and Preparation}


\subsection{Data Sources}


Two complementary datasets were collected from publicly available environmental data
repositories: historical air quality observations from the OpenAQ v3 API, and
meteorological observations from the Open-Meteo Historical Weather API.

\textbf{Air Quality Dataset.} Station discovery was performed automatically by querying all
monitoring stations located within a 25 km radius of central Kathmandu (Latitude:
27.7172$^\circ$, Longitude: 85.3240$^\circ$). For every discovered station, the corresponding PM$_{2.5}$
sensor ID was extracted, and hourly PM$_{2.5}$ concentration values with timestamps were
downloaded.

\subsection{Dataset Extraction Pipeline}


The extraction and preparation process was implemented as an automated, modular
pipeline of numbered, independent scripts, each reading from a defined input directory
and writing to a defined output directory.

All stages are orchestrated by a single driver script that runs every stage in order
and halts at the first failing stage, ensuring the pipeline cannot silently run out of
sequence.

\subsection{Dataset Description and Organization}


The collected dataset consists of hourly observations from 49 usable air quality
monitoring stations across Kathmandu Valley, each stored as an independent station-wise
CSV file, alongside a station metadata file recording geographical coordinates for use
in graph construction.

\subsection{Feature Description}


PM$_{2.5}$ is used as the prediction target. Meteorological variables (temperature,
humidity, dew point, pressure, wind speed, wind direction, precipitation) serve as
explanatory input features. Latitude and longitude are reserved for graph construction
rather than used as direct prediction inputs.

\subsection{Data Preprocessing}


\textbf{Dataset Validation.} An automated validation process checks file readability, row/column
counts, duplicate timestamps, missing values, and chronological ordering, producing
\texttt{weather\_validation.csv} and \texttt{air\_quality\_validation.csv}.

\textbf{Dataset Profiling.} Following validation, a profiling stage summarizes observation
counts, valid/missing PM$_{2.5}$ records, and overall coverage per station in
\texttt{station\_coverage.csv}, used to decide which stations proceed to merging.

\textbf{Timestamp Normalization and Merging.} Air-quality timestamps (timezone-aware) and
weather timestamps (timezone-naive) are converted to a common, timezone-free, hourly
format. For each station, air-quality data is \textbf{left-joined onto the weather timeline}
(weather as the base table), since Open-Meteo's historical record is continuous at
hourly resolution. This guarantees one row per hour in the merged dataset, with \texttt{pm2\_5}
left as \texttt{NaN} wherever no reading exists -- a property later relied upon for
row-position-based feature engineering.

\textit{Note: \texttt{coverage (\%)} reflects the proportion of the full weather timeline for which
PM$_{2.5}$ readings exist -- stations with 0\% coverage (e.g. Bagdol) have a weather record
but no associated air-quality sensor data, and are excluded from modeling.}

\textbf{Removal of Initial Missing PM$_{2.5}$ Records.} Leading rows before the first valid PM$_{2.5}$
reading are trimmed per station; internal gaps occurring later are preserved for the
target-invalidation logic described below.

\textbf{Timestamp Continuity Validation.} Immediately after trimming, and \textit{before} feature
engineering, an automated gap check computes the interval between every pair of
consecutive timestamps per station, flagging any interval that is not exactly one hour.
This step is placed before feature engineering deliberately, since the lag/rolling
features described next rely on row position corresponding to real elapsed time.

\textbf{Feature Engineering.} The following features are generated per station:

\begin{itemize}
    \item \textit{Lag features}: \texttt{lag\_1}, \texttt{lag\_3}, \texttt{lag\_6}, \texttt{lag\_12}, \texttt{lag\_24} (PM$_{2.5}$ at 1--24 hours prior)
    \item \textit{Rolling statistics}: rolling mean and rolling standard deviation over 3-, 6-, and 24-hour windows
    \item \textit{Cyclical time features}: \texttt{hour\_sin}/\texttt{hour\_cos}, \texttt{month\_sin}/\texttt{month\_cos}
    \item \textit{Wind vector components}: $$wind_u = WindSpeed \cdot \cos(\theta), \qquad wind_v = WindSpeed \cdot \sin(\theta)$$
\end{itemize}

\textbf{Dataset Preparation and Target Definition.} The prediction target is defined as
PM$_{2.5}$ one hour ahead: $target\_pm2\_5(t) = pm2\_5(t+1)$. A naive \texttt{.shift(-1)}
implementation is unsafe across timestamp gaps, since it grabs whichever row is next in
the file regardless of real elapsed time. To prevent mislabeled targets, the target is
invalidated whenever the true gap to the next row is not exactly one hour:

\begin{verbatim}
next_timestamp = df["timestamp"].shift(-1)
time_gap = next_timestamp - df["timestamp"]

df["target_pm2_5"] = df["pm2_5"].shift(-1)
df.loc[time_gap != pd.Timedelta(hours=1), 
"target_pm2_5"] = pd.NA
\end{verbatim}

Rows with an invalidated or otherwise missing target are dropped, and the number of
rows removed is logged per station.

\textbf{Train / Validation / Test Split.} The prepared dataset was split chronologically
(not randomly) per station: 70\% training, 15\% validation, 15\% testing, to prevent
information leakage from future observations and reflect a realistic forecasting
scenario.

\textbf{Split Verification.} An automated check confirms correct sample counts,
chronological ordering, absence of overlapping timestamps, and successful three-way
separation before any station's data is used for model training.

\section{Baseline Forecasting Models}


\subsection{Persistence Baseline}


Predicts next-hour PM$_{2.5}$ as equal to the current reading: $\hat{y}_{t+1} = pm2\_5(t)$.
Despite its simplicity, this is a strong baseline for short-horizon PM$_{2.5}$ forecasting,
since pollution levels change relatively little hour to hour.

\subsection{Linear (Ridge) Regression Baseline}


\textbf{Initial attempt and multicollinearity issue.} The first implementation used ordinary
least squares (OLS) with every engineered column as input. This produced an average
test $R^2$ of \textbf{-2.98}, substantially worse than persistence, with some station
predictions falling far outside the physically plausible PM$_{2.5}$ range.

A feature correlation analysis revealed severe multicollinearity among the engineered
PM$_{2.5}$-derived columns:

With input columns this collinear, the design matrix for several stations had a
condition number on the order of $10^{16}$--$10^{18}$ (a well-conditioned matrix is
typically $10^2$--$10^4$), causing unstable OLS coefficients and unbounded predictions.

\textbf{Fix.} Two changes were made: (1) switching from plain \texttt{LinearRegression} to
\texttt{Ridge(alpha=10.0)}, which penalizes large coefficients; and (2) pruning the feature set
to one representative per group of near-duplicate columns, selected by correlation with
the target rather than dropped arbitrarily:

direction removed.

\section{Wind-Aware Graph Construction (In Progress)}


Development of the wind-aware dynamic graph module has begun as a separate pipeline.

\textbf{Completed:}
\begin{itemize}
    \item \textbf{Station node mapping} -- assigns each station a numeric graph node ID from its metadata, producing a station-to-ID lookup used by all subsequent graph steps.
    \item \textbf{Distance matrix generation} -- computes the pairwise Haversine (great-circle) distance between every pair of stations, producing a full distance matrix and a flattened edge list. This implements the spatial-proximity term $f_d(i,j)$.
\end{itemize}

\textbf{Remaining:}
\begin{itemize}
    \item Bearing matrix (compass direction between station pairs, needed to compare against wind direction)
    \item Static graph assembly (distance + historical PM$_{2.5}$ correlation)
    \item Dynamic wind-aware edge weights (time-varying term $f_w$)
    \item Graph snapshot generation (one adjacency matrix per timestamp)
    \item Sliding window construction for temporal model input
\end{itemize}
